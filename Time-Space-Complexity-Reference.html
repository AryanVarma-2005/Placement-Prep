01

Why Complexity Analysis Matters

Before learning how to calculate complexity, it helps to understand exactly what question it answers — and why "just run it and see" isn't good enough.

Every algorithm makes a promise: give me an input, and I'll produce an output. Complexity analysis answers a different question — not "what does it output," but "what does it cost to get there, and how does that cost change as the input grows?"

There are two costs we almost always care about:

  • Time complexity — how the number of basic operations grows as input size n grows.
  • Space complexity — how the amount of extra memory used grows as n grows.

Why not just measure runtime in seconds?

You could run an algorithm and time it with a stopwatch — but that number is almost useless for comparing algorithms, because it depends on:

  • The speed of the specific CPU it ran on
  • The programming language and its runtime overhead
  • What else was running on the machine at the time
  • Compiler/interpreter optimizations
  • The exact input used, not the general behavior across all inputs

None of these tell you anything fundamental about the algorithm itself. Complexity analysis strips all of that away and asks a hardware-independent question: as the input gets larger and larger, how does the workload scale?

💡
The core idea Two algorithms solving the same problem can both be "correct," but one might take milliseconds and the other might take longer than the universe has existed once n gets large enough. Complexity analysis is how you predict which is which before you run anything.

A concrete example

Suppose you need to check whether a list contains duplicate values. Two approaches:

Python
# Approach A — compare every pair
def has_duplicate_A(arr):
    n = len(arr)
    for i in range(n):
        for j in range(i + 1, n):
            if arr[i] == arr[j]:
                return True
    return False

# Approach B — use a set to remember what we've seen
def has_duplicate_B(arr):
    seen = set()
    for x in arr:
        if x in seen:
            return True
        seen.add(x)
    return False

Approach A is O(n²) — for every element, it scans nearly the whole rest of the list. Approach B is O(n) — it looks at each element exactly once. For a list of 1,000 items, that's roughly 1,000,000 comparisons versus 1,000 lookups. At 1,000,000 items, it's 10¹² operations versus 10⁶ — the difference between a program that finishes instantly and one that doesn't finish in a reasonable lifetime.

This is the entire reason complexity analysis exists: it lets you see this gap before you write or run a single line of production code, just by reasoning about the algorithm's structure.

What this guide covers

We'll build up from first principles — counting operations by hand — to formal notation, then to a repeatable set of rules for analyzing loops, recursion, and data structure operations, and finally to a fast pattern-recognition checklist you can apply to almost any code you see.

Asymptotic Notation: Big-O, Big-Ω, Big-Θ
02

Asymptotic Notation

Big-O, Big-Omega, and Big-Theta are the formal language for describing growth. Here's precisely what each one means — and why Big-O gets used so much more often than the others.

"Asymptotic" means as the input size approaches infinity. We're not interested in exact operation counts (those depend on hardware and constants) — we're interested in the shape of the growth curve as n gets arbitrarily large.

Big-O notation — O(f(n)): the upper bound

Big-O describes the worst-case growth rate — an upper bound on how fast the runtime (or space) can grow. Formally:

∑
Definition T(n) = O(f(n)) if there exist positive constants c and n₀ such that T(n) ≤ c·f(n) for all n ≥ n₀.

In plain language: "once n is big enough, T(n) never grows faster than some constant multiple of f(n)." Big-O gives you a guarantee — your algorithm will never be worse than this bound, no matter how the input is arranged.

This is why Big-O dominates everyday usage: engineers care most about the worst case, because that's the scenario you must be prepared for.

Big-Omega notation — Ω(f(n)): the lower bound

∑
Definition T(n) = Ω(f(n)) if there exist positive constants c and n₀ such that T(n) ≥ c·f(n) for all n ≥ n₀.

Big-Omega is the mirror image of Big-O — it describes a best-case floor. For example, any comparison-based sort must examine each element at least once, so comparison sorts are Ω(n) at minimum.

Big-Theta notation — Θ(f(n)): the tight bound

∑
Definition T(n) = Θ(f(n)) if T(n) = O(f(n)) and T(n) = Ω(f(n)) — the upper and lower bounds match.

Theta means the algorithm's growth is exactly f(n) (up to constant factors) in all cases — not just bounded above, but pinned down precisely. When people say "this algorithm runs in O(n log n)" they often technically mean Θ(n log n) — the bound is tight, not just an upper limit.

Little-o and little-omega (a quick mention)

These describe strict (non-tight) bounds — o(f(n)) means strictly slower-growing than f(n), with no constant making them equal. You'll see these far less often in practice; Big-O, Big-Ω, and Big-Θ cover the vast majority of real analysis.

NotationBound typePlain meaningTypical use
O(f(n))Upper bound"Never slower than this"Worst-case guarantees (most common)
Ω(f(n))Lower bound"Never faster than this"Best-case / proving a problem's floor
Θ(f(n))Tight bound"Grows exactly like this"Precise characterization

Why we drop constants and lower-order terms

Say an algorithm does exactly 3n² + 5n + 12 operations. As n → ∞, the n² term completely dominates — at n = 1,000,000 it contributes 3,000,000,000,000 while the 5n term contributes a comparatively tiny 5,000,000. So we simplify:

  • Drop constants: 3n² becomes n² — the constant factor doesn't change the growth shape, only the steepness.
  • Drop lower-order terms: n² + n becomes n² — for large n, the smaller term becomes negligible.
  • Keep the dominant term: whichever term grows fastest is what survives in the final Big-O.
Simplification examples
5n³ + 100n² + 3n + 7        →   O(n³)
2^n + n^100                  →   O(2^n)      (exponential beats any polynomial)
n log n + n                  →   O(n log n)
0.0001 * n²                  →   O(n²)       (the tiny constant still loses to n eventually)
100000                       →   O(1)        (constant — doesn't depend on n at all)
⚠
Common misconception Big-O does not mean "worst case" by definition — it means "upper bound." But because engineers almost always state the upper bound of the worst case scenario, "Big-O" and "worst case" have become casually synonymous in everyday conversation. Technically you can have a Big-O for best case or average case too — it just describes which scenario you're bounding.

Counting Operations
03

Counting Operations — The Foundation

Before you can spot patterns instantly, you need to be able to do it the slow way: line by line, counting exactly what happens.

At its core, finding time complexity means answering: "as a function of input size n, how many basic operations does this code perform?" A basic operation is anything that takes constant time regardless of input size — an addition, a comparison, an assignment, an array index lookup.

Step-by-step: counting by hand

Python
def example(arr):
    total = 0                  # 1 operation
    n = len(arr)                # 1 operation

    for i in range(n):          # loop runs n times
        total += arr[i]         # 1 operation, runs n times → n operations

    if total > 0:                # 1 operation
        return total
    return 0

Adding it up: 1 + 1 + n + 1 = n + 3. As a Big-O bound, we drop the constant 3 (it doesn't scale with n), leaving O(n).

The key insight: only growth matters

Whether the function does n + 3 operations or n + 3000 operations, both are O(n) — because as n approaches infinity, that constant becomes irrelevant to the shape of the curve. This is precisely why Big-O is useful: it lets you ignore implementation noise and focus on structural growth.

What counts as "n"?

This sounds obvious but trips people up constantly. n is whatever variable represents the size of the input that's actually driving the work. Be careful when there are multiple inputs:

Python
def combine(list_a, list_b):
    for x in list_a:        # runs len(list_a) times
        print(x)
    for y in list_b:        # runs len(list_b) times
        print(y)

This is not O(n) for a single n — it's O(a + b) where a = len(list_a) and b = len(list_b). Collapsing two independent input sizes into one variable is a common analysis mistake (more on this in Chapter 14).

A practical shortcut: count the loop iterations

You rarely need to count every single statement. In practice, find the operations that are inside the deepest, most-repeated part of the code, count how many times that part executes as a function of n, and that's your dominant term.

Identify what's inside loops vs. outside loops

Code outside any loop runs once — it contributes a constant, which gets dropped. Code's true cost lives inside its loops and recursive calls.

Count how many times each loop body executes

A single loop from 0 to n runs n times. A nested loop multiplies the iteration counts of each level.

Multiply nested costs, add sequential costs

Two loops nested inside each other → multiply their bounds. Two loops one after another → add their bounds (then keep only the larger, since Big-O keeps the dominant term).

Express the total as a function of n, then simplify

Drop constants and lower-order terms to get the final Big-O class.

✓
Practical habit When reading unfamiliar code, scan for loops and recursive calls first — that's almost always where the complexity "lives." Straight-line code (no loops, no recursion) is always O(1) relative to input size, no matter how many lines it has.

Rules for Calculating Complexity
04

Rules for Calculating Time Complexity

A small set of combination rules covers almost every piece of code you'll ever analyze. Learn these once, apply them everywhere.

Rule 1 — Sequential statements: add them

When operations happen one after another (not nested), their costs add. Big-O then keeps only the largest term.

Python
def f(arr):
    for x in arr: print(x)        # O(n)
    for x in arr:
        for y in arr: print(x, y) # O(n^2)
    print("done")                  # O(1)

# total: O(n) + O(n^2) + O(1)  →  O(n^2)  (the n^2 term dominates)

Rule 2 — Nested loops: multiply them

When one loop runs inside another, multiply their iteration counts. This is the single most important rule for spotting O(n²) and worse.

Python
def f(arr):
    n = len(arr)
    for i in range(n):        # runs n times
        for j in range(n):    # runs n times, for EACH i
            print(arr[i], arr[j])
# total iterations: n * n = n^2  →  O(n^2)

Three nested loops over the same n → O(n³), and so on. Each additional independent nesting level multiplies in another factor of n.

Rule 3 — Conditionals: take the worst branch

For an if/else, Big-O (the upper bound) takes whichever branch costs more, since that branch could execute.

Python
def f(arr, target):
    if target in arr:        # O(n) membership check
        return "found"        # O(1)
    else:
        for x in arr:          # O(n)
            process(x)
# worst case across branches → O(n)

Rule 4 — Drop constants

Whether a loop runs n times or you run the same O(n) loop five separate times in a row (5n total), it's still O(n). The constant factor matters for real-world performance tuning, but not for the asymptotic class.

Rule 5 — Drop non-dominant terms

O(n² + n) simplifies to O(n²) because for large n, the n² term overwhelms the n term. Always keep only the fastest-growing term.

Rule 6 — Different input variables stay separate

As shown in Chapter 3, don't collapse two unrelated inputs into a single n. O(n·m) for nested loops over two different-sized collections is not the same as O(n²) unless you know n = m.

Rule 7 — Function calls: substitute their own complexity

If a loop body calls a function, you must account for that function's own complexity, not just treat the call as O(1).

Python
def contains(arr, val):        # O(n) — linear scan
    return val in arr

def f(arr):
    for x in arr:                # runs n times
        contains(arr, x)          # each call is O(n)
# total: n calls * O(n) per call = O(n^2), NOT O(n)

This is one of the most common real-world mistakes — calling a built-in like in, .sort(), .index(), or list.copy() inside a loop silently turns an O(n) loop into something far worse, because each call itself costs more than O(1).

Code patternCombination ruleResulting growth
Statement A; then statement BAdd: O(A) + O(B)Keep the larger
Loop A containing loop BMultiply: O(A) × O(B)Compounds
if (cost X) else (cost Y)Worst case: max(X, Y)Take the larger branch
Loop calling an O(f(n)) function n timesMultiply: n × O(f(n))Don't assume calls are O(1)
✓
The 90% rule Almost every analysis you'll do boils down to: find the loops, find their depth of nesting, find what each loop body actually costs (including any function calls), then add/multiply according to the rules above. The remaining chapters mostly add nuance to this core process.

The Big-O Catalog & Growth Chart

Every algorithm you'll encounter falls into one of a small number of growth families. Knowing their shapes — and how dramatically they diverge — is what makes complexity intuitive instead of abstract.

How the complexity classes diverge as n grows
Y-axis is logarithmic (each gridline is 1000× the one below) — on a normal scale, O(2ⁿ) and O(n!) would shoot off the page within a few steps. Click a legend item to toggle a curve.
O(1)

Notice how O(1) and O(log n) stay flat near the bottom no matter how large n gets, while O(n!) rockets past every other curve before n even reaches 15. This is the entire reason engineers obsess over complexity class — at small n, the difference is academic; at real-world scale, it's the difference between "instant" and "impossible."

The catalog, from best to worst

O(1) — Constantexcellent

Runtime doesn't depend on input size at all — one operation, or a fixed number of operations, regardless of how large n gets.

Examples: array index access arr[5], hash table insert/lookup (average case), push/pop on a stack, checking if a number is even.

O(log n) — Logarithmicexcellent

Runtime grows very slowly — each step eliminates a large fraction (usually half) of the remaining problem. Doubling n only adds one more step.

Examples: binary search, balanced binary search tree operations, finding an element's depth in a balanced tree.

O(n) — Lineargood

Runtime grows proportionally with input size — double the input, double the work. Every element gets touched roughly once.

Examples: linear search, traversing an array or linked list once, finding the max/min of an unsorted list.

O(n log n) — Linearithmicdecent

The signature of "divide, solve, then combine" algorithms — break the problem in half repeatedly (the log n part), but still touch every element at each level (the n part).

Examples: merge sort, heap sort, quicksort (average case), any algorithm with a sort as its bottleneck step.

O(n²) — Quadraticcaution

Every element is compared or combined with every other element. Doubling the input quadruples the work. Becomes painful past roughly 10,000–100,000 elements.

Examples: bubble sort, insertion sort, selection sort, naive duplicate detection, comparing all pairs in a list.

O(n^k) — Polynomialcaution

The general family that O(n²) and O(n³) belong to — k nested loops over n produce O(nᵏ). Usually only tractable for small k and moderate n.

Examples: O(n³) naive matrix multiplication, triple-nested-loop brute-force algorithms.

O(2ⁿ) — Exponentialbad

Each additional input element doubles the total work. Becomes impractical past roughly n = 30–40, even on fast hardware.

Examples: naive recursive Fibonacci, generating every subset of a set, brute-force solving the traveling salesman problem.

O(n!) — Factorialcatastrophic

Considering every possible ordering or permutation of the input. Becomes unusable past roughly n = 12–15.

Examples: generating every permutation of a list, brute-force traveling salesman by trying every route.

⚠
Sense of scale For n = 20: O(n) ≈ 20 operations, O(n²) ≈ 400, O(n log n) ≈ 86, O(2ⁿ) ≈ 1,048,576, and O(n!) ≈ 2,432,902,008,176,640,000. The jump from polynomial to exponential/factorial isn't a gentle increase — it's a different universe of cost.

Analyzing Loops in Detail
06

Analyzing Loops in Detail

Loops are where most real-world complexity lives. Here's how to handle every shape of loop you'll run into — including the tricky ones that don't look like simple O(n) or O(n²) at first glance.

Single loops

Python
for i in range(n):
    do_something(i)   # O(1) per call
# n iterations, O(1) each → O(n)

Straightforward: a loop that runs n times with constant work per iteration is O(n).

Loops with non-unit steps

Python
for i in range(0, n, 2):     # step of 2 → n/2 iterations
    do_something(i)
# still O(n) — the constant factor (1/2) gets dropped

Stepping by a fixed amount (2, 3, 10...) only changes the constant factor, not the growth class. Still O(n).

Loops that multiply or divide the counter — logarithmic

Python
i = 1
while i < n:
    do_something(i)
    i *= 2          # doubling each time
# how many times can you double 1 before passing n? log2(n) times
# → O(log n)

This is the telltale signature of O(log n): the loop variable grows (or shrinks) multiplicatively rather than additively. Binary search uses the inverse pattern — halving a search range each step.

Nested loops with the same bound — quadratic

Python
for i in range(n):
    for j in range(n):
        do_something(i, j)
# n * n → O(n^2)

Nested loops with a dependent (triangular) bound

Python
for i in range(n):
    for j in range(i, n):      # inner bound depends on i
        do_something(i, j)

This one trips people up: the inner loop doesn't run n times every time — when i=0 it runs n times, when i=1 it runs n-1 times, and so on down to 1. Total iterations:

Math
n + (n-1) + (n-2) + ... + 1  =  n(n+1)/2  =  O(n^2)

Even though each inner loop is "shorter" than n, the sum of a triangular series is still quadratic — the constant factor (½) gets dropped, but the n² growth class survives. This pattern shows up constantly: comparing every pair without duplicates, bubble sort's optimized version, etc.

Nested loops over independent collections

Python
for x in list_a:        # length n
    for y in list_b:     # length m
        compare(x, y)
# O(n * m) — keep them separate unless you know n == m

Sequential (back-to-back) loops

Python
for i in range(n):
    step_one(i)        # O(n)
for i in range(n):
    for j in range(n):
        step_two(i, j)  # O(n^2)
# O(n) + O(n^2) → O(n^2), the larger term dominates

A loop combined with a logarithmic inner loop — linearithmic

Python
for i in range(n):       # n iterations
    j = 1
    while j < n:           # log n iterations
        j *= 2
        process(i, j)
# n * log(n) → O(n log n)

This is exactly how merge sort behaves conceptually: an outer pass over all n elements at log n levels of division.

A loop that breaks early

💡
Early exits affect best/average case, not worst case A loop with a break or early return can finish faster on some inputs — but for Big-O (the worst-case upper bound), you still assume the loop might run all the way to completion, unless you have a specific guarantee otherwise. See Chapter 9 for how to separate best/worst/average case properly.

Recursion & Recurrence Relations
07

Recursion & Recurrence Relations

Recursive functions can't be analyzed by simply counting loop iterations — you need to express their cost as a recurrence relation, then solve it. Here are the three standard methods.

Step 1 — write the recurrence relation

A recurrence relation expresses the cost of solving a problem of size n in terms of the cost of solving smaller subproblems. The general form is:

Form
T(n) = a * T(n / b) + f(n)

a    = how many recursive calls are made per level
n/b  = the size of each subproblem (input shrinks by factor b)
f(n) = the cost of the work done outside the recursive calls
       (combining results, splitting the input, etc.)

Let's derive this for a few familiar algorithms:

Python — Binary Search
def binary_search(arr, lo, hi, target):
    if lo > hi:
        return -1
    mid = (lo + hi) // 2
    if arr[mid] == target:
        return mid
    elif arr[mid] < target:
        return binary_search(arr, mid + 1, hi, target)   # 1 recursive call
    else:
        return binary_search(arr, lo, mid - 1, target)    # on half the input
# T(n) = 1*T(n/2) + O(1)   → a=1, b=2, f(n)=O(1)
Python — Merge Sort
def merge_sort(arr):
    if len(arr) <= 1:
        return arr
    mid = len(arr) // 2
    left = merge_sort(arr[:mid])     # T(n/2)
    right = merge_sort(arr[mid:])     # T(n/2)
    return merge(left, right)          # O(n) to merge
# T(n) = 2*T(n/2) + O(n)   → a=2, b=2, f(n)=O(n)

Step 2 — solve it: three methods

Method 1 — The Recursion Tree

Draw the recursion as a tree: each node is a call, its children are the recursive calls it makes. Sum the work done at each level, then sum across all levels.

Merge sort recursion tree
Level 0:        T(n)                              work = n
Level 1:     T(n/2)  T(n/2)                        work = n/2 + n/2 = n
Level 2:   T(n/4) T(n/4) T(n/4) T(n/4)               work = 4*(n/4) = n
   ...
Level log2(n):  T(1) T(1) ... (n leaves)            work = n*1 = n

Number of levels = log2(n)
Work per level    = n  (constant across all levels)
Total work        = n * log2(n)  →  O(n log n)

The recursion tree method is the most intuitive — it directly visualizes "how much work happens at each depth," which is exactly what determines the final complexity.

Method 2 — The Master Theorem

For recurrences of the form T(n) = a·T(n/b) + f(n), the Master Theorem gives a direct formula without needing to draw anything. Compare f(n) against n^(log_b a) — the "critical exponent":

CaseConditionResult
Case 1f(n) = O(n^(log_b a − ε)) for some ε > 0 — the recursive calls dominateT(n) = Θ(n^(log_b a))
Case 2f(n) = Θ(n^(log_b a)) — work is balanced at every levelT(n) = Θ(n^(log_b a) · log n)
Case 3f(n) = Ω(n^(log_b a + ε)) — the combine step dominatesT(n) = Θ(f(n))
Worked example — Merge Sort
T(n) = 2*T(n/2) + O(n)
a = 2, b = 2, f(n) = n

n^(log_b a) = n^(log2 2) = n^1 = n
f(n) = n, which equals n^(log_b a)  →  this is Case 2

Result: T(n) = Θ(n^1 * log n) = Θ(n log n)   ✓ matches the tree method
Worked example — Binary Search
T(n) = 1*T(n/2) + O(1)
a = 1, b = 2, f(n) = 1 (constant)

n^(log_b a) = n^(log2 1) = n^0 = 1
f(n) = O(1), which equals n^(log_b a)  →  Case 2

Result: T(n) = Θ(1 * log n) = Θ(log n)   ✓
⚠
Master Theorem limitations It only applies cleanly to the form a·T(n/b) + f(n) with constant a and b. Recurrences like T(n) = T(n-1) + T(n-2) (Fibonacci) or with non-polynomial f(n) need the recursion tree or substitution method instead.

Method 3 — Substitution (guess and verify)

Guess the closed-form answer, then prove it by induction. Useful when the Master Theorem doesn't apply.

Example
T(n) = T(n - 1) + O(1)     (e.g. a simple linear recursive scan)

Guess: T(n) = O(n)
Verify: T(n) = T(n-1) + c = [c(n-1)] + c = cn   ✓ confirms O(n)

T(n) = T(n - 1) + O(n)     (e.g. recursive selection sort pass)

Guess: T(n) = O(n^2)
Verify: T(n) = T(n-1) + n. Expanding: n + (n-1) + (n-2) + ... + 1
       = n(n+1)/2 = O(n^2)   ✓

Multi-branch recursion — exponential blowup

When a function makes more than one recursive call without shrinking the problem enough — and especially when subproblems overlap — complexity explodes:

Python — naive Fibonacci
def fib(n):
    if n <= 1:
        return n
    return fib(n - 1) + fib(n - 2)
# T(n) = T(n-1) + T(n-2) + O(1)
# Two branches, each shrinking by only 1 → O(2^n)
# (technically closer to O(1.618^n), the golden ratio, but bounded by O(2^n))

This is also the textbook case for memoization — caching results of subproblems turns this from O(2ⁿ) into O(n), because each of the n unique subproblems then gets computed only once.

A repeatable process for recursive analysis

Write T(n) in terms of smaller subproblems

Identify how many recursive calls are made, how much smaller each subproblem is, and how much non-recursive work happens per call.

Choose a solving method

Recursion tree for intuition, Master Theorem for speed (when the form fits), substitution for anything unusual.

Check for overlapping subproblems

If the same subproblem gets recomputed many times (like naive Fibonacci), the real-world fix is memoization or dynamic programming — not just accepting the exponential bound.

Space Complexity in Detail
08

Space Complexity in Detail

Space complexity gets less attention than time, but the same systematic approach applies — and recursion makes it especially easy to get wrong.

What counts toward space complexity?

Space complexity measures the total memory an algorithm needs, as a function of input size n. It's useful to split this into two parts:

  • Input space — the memory used to store the input itself. This is usually excluded from the analysis, since you didn't choose to create that data — it was given to you.
  • Auxiliary space — the extra memory the algorithm allocates while running: temporary variables, helper arrays, hash maps, recursion call stack frames, etc.
💡
The convention When people say an algorithm "runs in O(1) space" or "is in-place," they almost always mean auxiliary space — not counting the input itself. Always check which convention a source is using before comparing numbers.

Counting auxiliary space — the same rules as time, applied to memory

Python
def sum_array(arr):
    total = 0              # one int variable → O(1) extra space
    for x in arr:
        total += x
    return total
# auxiliary space: O(1) — a fixed number of variables, regardless of len(arr)

def duplicate_array(arr):
    new_arr = []
    for x in arr:
        new_arr.append(x)   # grows with n
    return new_arr
# auxiliary space: O(n) — the new array scales with input size

The hidden cost: the recursion call stack

This is the single most common space-complexity mistake. Every recursive call adds a new stack frame — memory holding that call's local variables and return address — and that frame stays on the stack until the call returns.

Python
def sum_recursive(arr, i=0):
    if i == len(arr):
        return 0
    return arr[i] + sum_recursive(arr, i + 1)
# time:  O(n)  — n calls total
# space: O(n)  — n stack frames exist simultaneously at maximum recursion depth,
#                even though each individual frame only holds O(1) of its own data
⚠
Rule of thumb Recursive space complexity is generally O(maximum recursion depth), not O(number of total calls). A function with O(n) total calls but only O(log n) depth at any one time (like a balanced binary recursion) uses O(log n) stack space — not O(n).
Python — comparing depth vs. call count
def merge_sort(arr):
    if len(arr) <= 1:
        return arr
    mid = len(arr) // 2
    left = merge_sort(arr[:mid])
    right = merge_sort(arr[mid:])
    return merge(left, right)
# Total calls: O(n) across the whole run
# Max simultaneous depth: O(log n)  (the tree halves each level)
# Stack space: O(log n)
# BUT: merge_sort also allocates new sub-arrays at every level (arr[:mid], arr[mid:])
#      which adds O(n) auxiliary space for the temporary arrays themselves
# → overall space complexity: O(n)  (the array copies dominate, not the stack)

Time–space tradeoffs

Many optimizations trade one resource for the other. Recognizing this tradeoff is part of complexity literacy:

TechniqueTime effectSpace effect
Memoization / cachingO(2ⁿ) → O(n) (often)O(1) → O(n) (cache storage)
Hash map for lookupsO(n) → O(1) average per lookup+O(n) for the hash map
In-place sorting (e.g. quicksort)Comparable to merge sortO(log n) vs. merge sort's O(n)

✓
How to analyze space, step by step 1) List every variable, array, or structure the algorithm allocates beyond the input. 2) For each, determine how its size scales with n. 3) If the algorithm is recursive, separately account for maximum call-stack depth. 4) Sum it all, then simplify using the same dominant-term rules as time complexity.

Best, Worst & Average Case
09

Best, Worst & Average Case

The same algorithm can have wildly different costs depending on the input it receives. Here's how to reason about each scenario — and why "worst case" became the default.

The three scenarios

  • Best case — the minimum possible cost, achieved by the most favorable input. Usually written with Big-Omega (Ω), though often casually expressed with Big-O too.
  • Worst case — the maximum possible cost, achieved by the most unfavorable input. This is what Big-O conventionally describes.
  • Average case — the expected cost over a (typically uniform) distribution of all possible inputs. Usually expressed with Big-Theta (Θ) when it's a tight, well-defined average.

Worked example — linear search

Python
def linear_search(arr, target):
    for i, x in enumerate(arr):
        if x == target:
            return i
    return -1
ScenarioWhat input causes itComplexity
Best caseTarget is the first elementO(1)
Average caseTarget is randomly positionedO(n/2) → O(n)
Worst caseTarget is last, or not present at allO(n)

Worked example — quicksort (where the gap is dramatic)

Quicksort picks a pivot and partitions the array around it. Its performance depends entirely on how balanced that partition turns out to be.

ScenarioWhat input causes itComplexity
Best / average casePivot roughly splits the array in half each timeO(n log n)
Worst casePivot is always the smallest/largest element (e.g. already-sorted input with a naive pivot choice)O(n²)

This is exactly why production implementations use techniques like randomized pivot selection or median-of-three — they make the catastrophic worst case astronomically unlikely in practice, even though it remains technically possible.

Why worst case is usually what's quoted

💡
Reliability over optimism When you're building a system, you need to plan for the worst input it might realistically receive — not hope for the best one. A server that's fast on average but occasionally grinds to a halt on adversarial input is a real liability (and a real security concern — see "algorithmic complexity attacks"). That's why, absent other context, "the complexity of an algorithm" defaults to meaning its worst case.

When average case is what actually matters

For algorithms used repeatedly across huge numbers of random, everyday inputs — like quicksort in practice, or hash table operations — the average case is often the more useful number, because the true worst case may be vanishingly rare or actively defended against. Both numbers are worth knowing; report whichever is relevant to your context, and label it clearly.

Amortized Analysis
10

Amortized Analysis

Some operations are usually cheap but occasionally expensive. Amortized analysis answers: what's the true average cost per operation, over a long sequence of operations — guaranteed, not just typically?

The classic example: dynamic array (Python list) append

A dynamic array starts with some fixed capacity. Appending is O(1) — until it's full, at which point it must allocate a new, larger array and copy every existing element over, which is O(n).

Conceptual behavior
capacity = 4
append() append() append() append()   ← all O(1), array fills up
append()                                ← capacity exceeded!
   → allocate new array of size 8
   → copy all 4 existing elements      ← this single call costs O(n)
   → then add the new element
append() append() append()             ← O(1) again, until capacity 8 fills

If you only looked at that one expensive resize call, you might (wrongly) conclude appending is O(n). But that resize doesn't happen every call — it happens less and less often as the array grows, if the array grows by doubling.

Why doubling capacity is the key trick

The standard strategy: when the array is full, double its capacity (not just add a fixed amount). This makes resizes increasingly rare relative to the number of cheap appends that happen between them.

✓
The aggregate method, in plain terms To append n elements total, starting from capacity 1 and doubling each time you run out: resizes happen at sizes 1, 2, 4, 8, ... up to n — that's about log₂(n) resizes. The total cost of all those resizes combined is 1 + 2 + 4 + ... + n ≈ 2n (a geometric series). Add the n cheap O(1) appends themselves, and the total work for n appends is O(n) — which means the average (amortized) cost per append is O(n)/n = O(1).
Why fixed-increment growth fails
BAD strategy: grow capacity by a fixed +1 each time it's full
→ resizes happen on every single append once you exceed initial capacity
→ each resize copies an ever-larger array
→ total cost for n appends: 1 + 2 + 3 + ... + n = O(n^2)
→ amortized cost per append: O(n) — no better than not amortizing at all!

GOOD strategy: double capacity each time it's full
→ total cost for n appends: O(n)
→ amortized cost per append: O(1)

Three formal methods for amortized analysis

  • Aggregate method — sum the total cost of a sequence of n operations, then divide by n. (Used above.)
  • Accounting (banker's) method — assign each operation an artificial "charge." Cheap operations are overcharged, building up "credit" that pays for the occasional expensive operation.
  • Potential method — define a potential function over the data structure's state, and show that the actual cost plus the change in potential is bounded — a more mathematical generalization of the accounting method.

For almost all practical purposes, the aggregate method (sum total cost, divide by number of operations) is enough to reason about amortized cost intuitively.

Where else amortized analysis shows up

Structure / operationOccasional expensive opAmortized cost
Dynamic array append (Python list, Java ArrayList)Resize + copy: O(n)O(1)
Hash table insertRehashing on load factor threshold: O(n)O(1) average
Incrementing a binary counterA long carry chain: O(log n)O(1)
Union-Find with path compressionA long find-root chainO(α(n)) ≈ constant
⚠
Amortized ≠ average case These are easy to confuse but mean different things. Average case depends on a probability distribution over random inputs — it can still be bad for a specific unlucky sequence. Amortized is a guarantee over any sequence of operations on the worst-case inputs — there's no luck involved; the bound always holds when spread over enough operations.

How to Identify Complexity Fast

How to Identify Complexity Fast

Once the rules are second nature, you stop calculating and start recognizing. Here's the mental checklist experienced engineers run almost subconsciously when they look at code.

The visual pattern-matching cheat sheet

What you see in the codeLikely complexity
No loops, no recursion — just statements and conditionalsO(1)
Loop variable is multiplied/divided each iteration (e.g. i *= 2); or repeatedly halving a search rangeO(log n)
A single loop over the input, constant work per iterationO(n)
Two separate (sequential, not nested) single loopsO(n)
A loop combined with a sort, or a loop containing a halving/doubling inner loopO(n log n)
Two nested loops, both bounded by n (full or triangular)O(n²)
k nested loops, all bounded by nO(nᵏ)
Recursion that makes 2+ calls per level without shrinking enough, with overlapping subproblemsO(2ⁿ)
Generating all permutations / orderings of the inputO(n!)

The five-question checklist

1. Are there any loops or recursive calls?

If no — it's O(1) regardless of how many lines of code there are. Straight-line code never scales with n.

2. How many levels of nesting, and are they independent?

Sequential → add. Nested → multiply. Watch for hidden nesting inside function calls (Chapter 4, Rule 7) — a "single" loop calling an O(n) function inside it is secretly nested.

3. Does the loop variable change additively or multiplicatively?

Additive (i += 1) → linear contribution. Multiplicative (i *= 2 or halving a range) → logarithmic contribution.

4. If recursive — how many calls per level, and how much does the input shrink?

One call, input halved → O(log n). Two calls, input halved, O(n) combine work → O(n log n) (merge sort pattern). Two calls, input only reduced by 1 each time → O(2ⁿ) (naive Fibonacci pattern).

5. Are there hidden costs inside built-in operations?

Sorting inside a loop, in checks on a list (not a set) inside a loop, string concatenation in a loop, .copy() or slicing inside a loop — these silently multiply your complexity. Always ask: "what is the actual cost of this single line, not just whether it looks like one operation?"

✓
The fastest sanity check Pick a tiny, a medium, and a large value of n in your head (say 10, 1,000, 100,000) and ask "roughly how many total operations would this run?" If doubling n roughly doubles the work → O(n). If it roughly quadruples the work → O(n²). If it adds a small constant amount of extra work → O(log n). This rough mental simulation catches most analysis mistakes faster than working through formal rules.

Common Mistakes & Tricky Cases
14

Common Mistakes & Tricky Cases

A collection of the analysis traps that catch even experienced programmers off guard.

Mistake 1 — assuming built-in operations are O(1)

Python — looks O(n), is actually O(n²)
def has_dup_slow(arr):
    seen = []                  # a LIST, not a set
    for x in arr:
        if x in seen:           # 'in' on a list is O(n)!
            return True
        seen.append(x)
    return False
# n iterations * O(n) membership check each = O(n^2)
# Fix: use a set() instead — 'in' on a set is O(1) average → O(n) total

Other common O(n)-disguised-as-O(1) culprits: list.pop(0) (O(n), shifts every element — use collections.deque instead), string concatenation with += in a loop (each concatenation can copy the whole string), and list.count() or list.index() inside a loop.

Mistake 2 — collapsing two different input sizes into one n

Python
def f(list_a, list_b):
    for x in list_a:
        for y in list_b:
            compare(x, y)
# WRONG: "O(n^2)"
# RIGHT: O(a * b), where a = len(list_a), b = len(list_b)
# These are only equal if you specifically know a == b

Mistake 3 — confusing total recursive calls with stack depth

As covered in Chapter 8: a recursive function might make O(n) total calls but only ever have O(log n) of them active on the stack simultaneously (e.g. balanced divide-and-conquer). Time complexity counts total calls; space complexity (for the stack) counts maximum simultaneous depth — these are not the same number.

Mistake 4 — ignoring the cost of slicing/copying

Python
def f(arr):
    n = len(arr)
    for i in range(n):
        sub = arr[i:]      # slicing creates a NEW list — O(n) per call, not O(1)
        process(sub)
# n iterations * O(n) slice each = O(n^2) time, not O(n)

This exact trap shows up in merge sort implementations that slice arrays recursively (arr[:mid], arr[mid:]) — each slice is its own O(k) copy operation, which is part of why careful implementations sometimes pass index ranges instead of creating new sub-arrays.

Mistake 5 — treating an exponential algorithm with a small constraint as "fine"

O(2ⁿ) looks scary in the abstract, but if a problem guarantees n ≤ 20 (common in some constraint-based problems), 2²⁰ ≈ 1 million — entirely fine. Complexity class alone doesn't tell you whether something is fast enough; you need it combined with the actual expected input size.

Mistake 6 — forgetting that two loops can hide a third, implicit one

Python — looks O(n), is O(n^2)
def flatten_and_sum(matrix):     # matrix: list of lists
    total = 0
    for row in matrix:             # n rows
        for val in row:              # m values per row
            total += val
# this is O(n*m), often described loosely as O(n) when really
# "n" here secretly means "total number of elements across the whole matrix"
# — always clarify what n actually refers to before quoting a bound

Mistake 7 — assuming sorted input is "free"

If a problem statement gives you an unsorted array and your solution requires it sorted first, the O(n log n) sort cost is part of your total complexity — don't silently drop it because "sorting is a separate step." Add it like any other sequential operation (Chapter 4, Rule 1).

✕
The meta-mistake The single biggest source of complexity-analysis errors is treating any line of code as "free" without checking what it actually does under the hood. Every operation has a real cost — the skill is knowing (or looking up) what that cost is, not assuming.

Practice Problems
15

Practice Problems

Work out each one yourself before expanding the solution. These are ordered roughly by difficulty.

Problem 1 — A single loop with a conditional
Python
def f(arr):
    count = 0
    for x in arr:
        if x % 2 == 0:
            count += 1
    return count

Solution: One loop over n elements, O(1) work per iteration (a modulo check and maybe an increment). Total: O(n) time, O(1) space (just the count variable).

Problem 2 — Nested loops with a dependent bound
Python
def f(arr):
    n = len(arr)
    pairs = []
    for i in range(n):
        for j in range(i + 1, n):
            pairs.append((arr[i], arr[j]))
    return pairs

Solution: Triangular nested loop — total iterations = n(n-1)/2 → O(n²) time. Space is also O(n²), since the pairs list stores roughly n²/2 tuples — don't forget the output itself counts toward space complexity when it scales with n.

Problem 3 — A while loop that halves
Python
def f(n):
    steps = 0
    while n > 1:
        n = n // 2
        steps += 1
    return steps

Solution: n is halved each iteration — classic logarithmic signature. O(log n) time, O(1) space.

Problem 4 — Two sequential operations, one involving a sort
Python
def f(arr):
    arr.sort()                  # Timsort: O(n log n)
    total = 0
    for x in arr:                 # O(n)
        total += x
    return total

Solution: Sequential operations add: O(n log n) + O(n). Keep the dominant term → O(n log n) time. Python's Timsort uses O(n) auxiliary space in the worst case, so overall space is O(n).

Problem 5 — A loop hiding a linear search
Python
def f(arr1, arr2):
    common = []
    for x in arr1:
        if x in arr2:        # 'in' on a list: O(len(arr2))
            common.append(x)
    return common

Solution: With n = len(arr1) and m = len(arr2), this is O(n) iterations × O(m) per check = O(n·m) time — not O(n). Converting arr2 to a set first would bring it down to O(n + m), at the cost of O(m) extra space for the set.

Problem 6 — Recursive function: find the recurrence and solve it
Python
def f(arr, lo, hi):
    if lo >= hi:
        return
    mid = (lo + hi) // 2
    f(arr, lo, mid)
    f(arr, mid + 1, hi)
    for i in range(lo, hi):       # O(n) work at this level
        touch(arr[i])

Solution: T(n) = 2T(n/2) + O(n) — exactly the merge sort recurrence (a=2, b=2, f(n)=n). By the Master Theorem, Case 2 applies (f(n) = n^(log₂2) = n) → O(n log n) time. Max recursion depth is O(log n) → O(log n) auxiliary stack space.

Problem 7 — Multiple independent loops plus a nested one
Python
def f(arr):
    n = len(arr)
    for x in arr: print(x)                       # O(n)
    total = sum(arr)                                # O(n) — sum() scans the list
    for i in range(n):
        for j in range(n):
            if arr[i] + arr[j] == total:               # O(1) per pair
                print("match")
    return total

Solution: O(n) + O(n) + O(n²) — the nested loop dominates. Total: O(n²) time, O(1) auxiliary space (no structure grows with n beyond fixed variables).

Problem 8 — Spot the exponential blowup, and fix it
Python
def count_paths(n):
    if n <= 1:
        return 1
    return count_paths(n - 1) + count_paths(n - 2)

Solution: Same shape as naive Fibonacci: T(n) = T(n-1) + T(n-2) + O(1), two branches each shrinking by a small constant, heavy subproblem overlap → O(2ⁿ) time, O(n) space (max recursion depth, since one branch always recurses down to the base case before backtracking).

The fix: memoize results by n (a dictionary or array cache). Each of the n unique subproblems then computes once → O(n) time, O(n) space for the cache plus the call stack.
