Chapter 01

Introduction to Git

Beginner

Git is a free, open-source distributed version control system created by Linus Torvalds in 2005. Unlike centralized VCS tools, every developer has a full copy of the repository — including complete history — on their local machine.

Why Git?

🔀 Distributed

Every clone is a full backup. No single point of failure. Work completely offline.

⚡ Fast

Most operations are local. No network round-trips for diff, log, branch, or checkout.

🌿 Branching

Lightweight branches make experimenting risk-free. Create, merge, delete in seconds.

🔐 Integrity

Every file and commit is checksummed with SHA-1. History cannot be altered silently.

The Three States

ModifiedFiles changed but not yet staged
StagedFiles marked to go into next commit (index)

Global Configuration

Config 
# Set identity (required for every commit)

Chapter 04

Staging & Commits

Beginner

Staging Files

Shell 
# Stage a specific file
git add README.md

# Stage all changes in current directory
git add .

# Stage all tracked files with modifications
git add -u

# Stage everything (new + modified + deleted)
git add -A

# Interactively stage hunks (partial file staging)
git add -p src/app.js

# Unstage a file (keep changes in working dir)
git restore --staged README.md
# Older syntax
git reset HEAD README.md

Committing

Shell 
# Commit with a message
git commit -m "feat: add user authentication"

# Stage all tracked changes and commit in one step
git commit -am "fix: correct off-by-one in pagination"

# Open editor for detailed message
git commit

# Amend last commit (change message or add forgotten files)
git commit --amend -m "fix: correct off-by-one error in pagination"
git commit --amend --no-edit   # keep same message

# Empty commit (useful for CI triggers)
git commit --allow-empty -m "chore: trigger CI"

Conventional Commits (Industry Standard)

📝
Format: <type>[optional scope]: <description>

Types: feat, fix, docs, style, refactor, perf, test, build, ci, chore, revert
Commit Messages 
# Feature with scope
feat(auth): add JWT refresh token rotation

# Bug fix
fix(api): handle null pointer in user lookup

# Breaking change
feat!: remove deprecated v1 endpoints

# With body and footer
feat(cart): implement quantity discount logic

Apply tiered discounts when cart quantity > 10.
Discounts are configurable per product category.

BREAKING CHANGE: CartItem now requires `category` field
Closes #142

Chapter 05

Branching

Intermediate

Branches are lightweight movable pointers to commits. Creating a branch is nearly instant and costs almost nothing in Git.

Branch Operations

Shell 
# List branches
git branch               # local branches
git branch -r           # remote branches
git branch -a           # all branches
git branch -v           # with last commit
git branch --merged     # branches merged into current
git branch --no-merged  # branches not yet merged

# Create a new branch
git branch feature/user-profile

# Create and switch (modern)
git switch -c feature/user-profile

# Create and switch (classic)
git checkout -b feature/user-profile

# Switch to existing branch
git switch main
git checkout main

# Rename a branch
git branch -m old-name new-name

# Delete branches
git branch -d feature/user-profile  # safe delete (merged only)
git branch -D feature/user-profile  # force delete
git push origin --delete feature/user-profile  # delete remote

Branch Strategy Visualization

main ●────────────────────────────────● \ / develop ●──────●────────────────●──────● \ / feature ●──●──●──●──● hotfix main ●─────────────● (urgent patch)

Chapter 06

Merging & Rebasing

Intermediate

Merging

Shell 
# Merge feature branch into current branch
git checkout main
git merge feature/user-profile

# Fast-forward merge (default if no divergence)
git merge --ff-only feature/quick-fix

# Always create a merge commit (no fast-forward)
git merge --no-ff feature/user-profile

# Squash all commits from branch into one
git merge --squash feature/messy-work
git commit -m "feat: add user profile (squashed)"

# Abort a merge in progress
git merge --abort

Resolving Merge Conflicts

Conflict Resolution 
# Conflict markers in a file:
<<<<<<< HEAD
  console.log('version from main');
=======
  console.log('version from feature branch');
>>>>>>> feature/user-profile

# 1. Edit the file to resolve
# 2. Stage the resolved file
git add src/app.js

# 3. Complete the merge
git commit

# Use a merge tool
git mergetool                    # opens configured tool
git mergetool --tool=vimdiff

Rebasing

⚠️
Golden Rule: Never rebase commits that have been pushed to a shared remote branch. Rebasing rewrites history.
Shell 
# Rebase feature branch onto main
git checkout feature/user-profile
git rebase main

# Interactive rebase — edit last N commits
git rebase -i HEAD~5

# Interactive rebase from a specific commit
git rebase -i abc1234

# In interactive mode, commands per commit:
# pick   = use commit as-is
# reword = use commit but edit message
# edit   = pause and amend commit
# squash = meld into previous commit
# fixup  = squash, discard this message
# drop   = remove commit entirely

# Continue / abort rebase
git rebase --continue
git rebase --abort
git rebase --skip

Chapter 07

Remote Repositories

Intermediate

Managing Remotes

Shell 
# List remotes
git remote
git remote -v            # with URLs

# Add a remote
git remote add origin https://github.com/user/repo.git
git remote add upstream https://github.com/original/repo.git

# Rename / remove remote
git remote rename origin github
git remote remove upstream

# Change remote URL
git remote set-url origin git@github.com:user/repo.git

Fetch, Pull & Push

Shell 
# Fetch all remotes (download, don't integrate)
git fetch
git fetch origin
git fetch --all          # all remotes
git fetch --prune        # remove stale remote-tracking refs

# Pull = fetch + merge (or rebase)
git pull
git pull origin main
git pull --rebase        # rebase instead of merge
git pull --ff-only       # fast-forward only, fail otherwise

# Push to remote
git push origin main
git push -u origin feature/user-profile  # set upstream
git push                 # push to tracked upstream
git push --force-with-lease              # safe force push
git push --tags          # push all tags
🚫
Never use git push --force on shared branches. Use --force-with-lease instead — it fails if someone else has pushed, preventing data loss.

Tracking Branches

Shell 
# Check out a remote branch and track it
git checkout --track origin/feature/login
# Shorthand (Git infers tracking)
git checkout feature/login

# Show tracking information
git branch -vv

Stash & Tags

Intermediate

Stashing Work

Shell 
# Stash current changes
git stash
git stash push -m "WIP: auth middleware"

# Include untracked files
git stash push -u
git stash push --include-untracked

# List all stashes
git stash list

# Apply latest stash (keeps stash)
git stash apply
git stash apply stash@{2}

# Apply and remove latest stash
git stash pop

# Drop a stash
git stash drop stash@{1}
git stash clear          # delete all stashes

# Show stash diff
git stash show -p

# Apply stash to a new branch
git stash branch new-feature stash@{0}

Tags

Shell 
# Lightweight tag
git tag v1.0.0

# Annotated tag (recommended for releases)
git tag -a v1.0.0 -m "Release version 1.0.0"

# Tag a specific commit
git tag -a v0.9.0 abc1234 -m "Beta release"

# List tags
git tag
git tag -l "v1.*"

# Push tags to remote
git push origin v1.0.0
git push origin --tags

# Delete a tag
git tag -d v1.0.0
git push origin --delete v1.0.0

Chapter 09

Undoing Changes

Advanced

Restore & Reset

Shell 
# Discard unstaged changes in a file
git restore src/app.js
git checkout -- src/app.js  # classic syntax

# Discard ALL unstaged changes
git restore .

# Unstage a file
git restore --staged src/app.js

# ── git reset ──────────────────────────────────
# Soft: moves HEAD, keeps staged + working dir
git reset --soft HEAD~1

# Mixed (default): moves HEAD, unstages changes
git reset HEAD~1
git reset --mixed HEAD~1

# Hard: moves HEAD, DISCARDS all changes ⚠️
git reset --hard HEAD~1
git reset --hard origin/main

Revert (Safe History Rewrite)

Shell 
# Create a new commit that undoes a commit (safe for shared)
git revert abc1234
git revert HEAD             # revert last commit
git revert HEAD~3..HEAD      # revert last 3 commits
git revert --no-commit HEAD  # revert without auto-committing

Reflog — Your Safety Net

💡
Reflog tracks every position HEAD has been at for the last 90 days. It's your ultimate undo for almost any Git mistake.
Shell 
# View the reflog
git reflog
git reflog show main

# Recover from a bad reset
git reset --hard HEAD@{3}

# Recover a deleted branch
git checkout -b recovered-branch HEAD@{5}

Chapter 10

History & Log

Advanced

Git Log

Shell 
# Default log
git log

# Compact one-line per commit
git log --oneline

# Graphical branch visualization
git log --oneline --graph --decorate --all

# Filter by author
git log --author="Jane Doe"

# Filter by date
git log --since="2024-01-01" --until="2024-06-01"

# Search commit messages
git log --grep="auth"

# Search for code changes (pickaxe)
git log -S "getUserById"   # string
git log -G "get.*ById"    # regex

# Show diff for each commit
git log -p
git log -p --follow src/utils.js  # track renames

# Custom format
git log --pretty=format:"%h %an %ar - %s"

# Commits that changed a file
git log -- src/auth/login.js

Blame & Bisect

Shell 
# Show who last modified each line
git blame src/app.js
git blame -L 20,40 src/app.js  # lines 20-40 only

# ── Bisect: binary search for bug ──────────────
git bisect start
git bisect bad                    # current commit is broken
git bisect good v1.2.0            # last known good

# Git checks out middle commit — test it:
# If bad:  git bisect bad
# If good: git bisect good
# Repeat until Git finds the culprit commit

git bisect reset                  # restore HEAD when done

# Automated bisect with a test script
git bisect run npm test

Cherry-Pick

Shell 
# Apply a specific commit to current branch
git cherry-pick abc1234

# Cherry-pick a range of commits
git cherry-pick abc1234..def5678

# Cherry-pick without auto-committing
git cherry-pick --no-commit abc1234

# Continue / abort cherry-pick
git cherry-pick --continue
git cherry-pick --abort

Filter-Branch / Filter-Repo

⚠️
These operations rewrite history. Only use on repositories you fully control. Always back up first.
Shell 
# Remove a file from entire history (use git-filter-repo — preferred)
pip install git-filter-repo
git filter-repo --path secrets.txt --invert-paths

# Extract a subdirectory as a new repo
git filter-repo --subdirectory-filter src/

Git Hooks

Hooks are scripts in .git/hooks/ that run automatically at key Git events.

bash · .git/hooks/pre-commit 
#!/bin/sh
# Run linter before every commit
npm run lint
if [ $? -ne 0 ]; then
  echo "Lint failed. Commit aborted."
  exit 1
fi
HookTriggerCommon Uses
pre-commitBefore commit is createdLinting, formatting, tests
commit-msgAfter commit message enteredEnforce commit message format
pre-pushBefore git pushRun test suite
post-mergeAfter successful mergeInstall dependencies
pre-rebaseBefore rebasingBlock rebasing on certain branches

Sparse Checkout

Shell 
# Checkout only part of a large monorepo
git clone --filter=blob:none --sparse https://github.com/big/monorepo.git
cd monorepo
git sparse-checkout init --cone
git sparse-checkout set packages/frontend packages/shared

Advanced Diff & Patch

Shell 
# Create a patch file
git diff > my-changes.patch
git format-patch HEAD~3         # 3 patch files, one per commit
git format-patch main..feature  # all commits on feature vs main

# Apply a patch
git apply my-changes.patch
git am 0001-my-feature.patch     # apply with commit info

Industry Git Workflows

Expert

Git Flow

A strict branching model designed for scheduled releases. Suitable for products with versioned releases.

main ●───────────────────────────────────● ↑ ↑ (merge + tag) develop ●──●──────────●──────────────●──────● ↑ ↑ ↑ feature ●─●─● ●─●─● ●─● hotfix main ●───────────● ↓ (merged to main AND develop)
BranchPurposeMerges Into
mainProduction-ready code—
developIntegration branch for featuresmain (via release)
feature/*New feature developmentdevelop
release/*Release preparation & bugfixesmain + develop
hotfix/*Critical production patchesmain + develop

GitHub Flow

Simpler workflow built for continuous delivery. Everything branches off main.

  1. Create a branch

    Branch off main with a descriptive name: feature/add-oauth, fix/null-pointer-checkout

  2. Add commits

    Make small, focused commits with clear messages. Keep the branch short-lived.

  3. Open a Pull Request

    Push branch and open a PR. Describe what changed and why. Link to issues.

  4. Review & Discuss

    Team reviews code, leaves comments, requests changes. Iterate.

  5. Merge & Deploy

    Merge to main. CI/CD deploys automatically. Delete the branch.

Trunk-Based Development

Engineers commit directly to main (or very short-lived branches, < 1 day). Relies heavily on feature flags and CI/CD. Preferred at Google and Meta scale.

TBD Commands 
# Create very short-lived branch
git switch -c feature/oauth-login

# Rebase frequently onto trunk
git fetch origin
git rebase origin/main

# Merge back quickly (same day)
git switch main
git merge --squash feature/oauth-login
git commit -m "feat(auth): add OAuth login (behind feature flag)"
git push
